# ASON Specification v1.0

> **ASON** = Array-Schema Object Notation  
> _"The efficiency of arrays, the structure of objects."_

ASON is a serialization format designed for large-scale data exchange and LLM interaction. This page is the normative specification. All language implementations must conform to the rules defined here.

## 1. Design Principles

- **Schema & data separation** — structure is declared once, never repeated in data rows
- **Row-oriented** — multiple records appear as consecutive tuples after a single schema header
- **Structural harmony** — `{}` defines the skeleton (schema), `()` holds the data (values)
- **Token-optimal** — minimal syntax characters; 30–70% fewer tokens than equivalent JSON
- **Extreme parse performance** — zero key-hashing, schema-driven parsing, low memory footprint

## 2. Formal Grammar

```abnf
document    = schema ":" SP data
schema      = "{" field-list "}"
field-list  = field *("," SP field)
field       = name [":" type]
name        = 1*(ALPHA / DIGIT / "_")
type        = primitive / schema / "[" type "]"
primitive   = "int" / "float" / "str" / "bool"

data        = tuple / tuple-list
tuple-list  = tuple *("," NL SP tuple)
tuple       = "(" value-list ")"
value-list  = value *("," value)
value       = scalar / tuple / array / ""
array       = "[" [value-list] "]"

scalar      = integer / float / boolean / quoted-str / unquoted-str
integer     = ["-"] 1*DIGIT
float       = ["-"] 1*DIGIT "." 1*DIGIT [("e"/"E") ["+"/"-"] 1*DIGIT]
boolean     = "true" / "false"
quoted-str  = DQUOTE *char DQUOTE
unquoted-str = 1*safe-char
safe-char   = any Unicode except "," / "(" / ")" / "[" / "]" / "\"
```

## 3. Data Types

### 3.1 Integer
Any value in the `i64` range (`-9223372036854775808` to `9223372036854775807`). Written as decimal digits with optional leading `-`. No leading zeros.

### 3.2 Float
IEEE 754 double-precision. Written as `[-]digits.digits` with optional `e` / `E` exponent. Values `NaN` and `Inf` are not representable in ASON Text (use `null` / empty slot instead).

### 3.3 Boolean
Exactly `true` or `false`. Case-sensitive, lowercase only.

### 3.4 Null / None
An empty slot — zero characters between two commas, or between the last comma and the closing `)`. Every implementation must map this to the language-native null/option/pointer type.

### 3.5 Unquoted String
Any sequence of Unicode code points that does not contain `,`, `(`, `)`, `[`, `]`, or `\`. Leading and trailing ASCII whitespace (space, tab) is **stripped** by the parser.

### 3.6 Quoted String
Characters enclosed in double quotes `"`. Supports escape sequences:

| Escape | Unicode |
|--------|---------|
| `\\` | U+005C |
| `\"` | U+0022 |
| `\n` | U+000A |
| `\t` | U+0009 |
| `\r` | U+000D |
| `\,` | U+002C (unquoted context only) |

### 3.7 Array
Comma-separated values inside `[` `]`. May be empty (`[]`). Elements follow the same `value` rules.

### 3.8 Nested Struct
A nested struct schema appears inline in the parent schema:

```ason
{id:int, address:{city:str, zip:str}}
```

Its data is an inner tuple:

```ason
(1, (Berlin, 10115))
```

## 4. Canonicalization

When a ASON document is generated by a conforming tool, it SHOULD:

1. Include type annotations in the schema (`{id:int, name:str}`)
2. Separate each tuple onto its own line, indented by two spaces
3. Separate the schema line and the first data tuple with a newline after `:`

```ason
[{id:int, name:str, active:bool}]:
  (1, Alice, true),
  (2, Bob,   false)
```

Non-canonical whitespace is valid for parsing.

## 5. Single-Struct Shorthand

A document containing exactly one tuple MAY omit the newline:

```ason
{id:int, name:str}:(1, Alice)
```

## 6. Binary Format (ASON-BIN)

ASON-BIN is a separate encoding that does not use the text schema syntax. Fields are laid out in declaration order with no padding. All integers are **little-endian**.

See the [Binary Format guide](/guide/binary-format) for the complete wire-encoding table.

## 7. Versioning

This document describes **ASON v1.0**. Future versions will increment the minor number for backward-compatible additions and the major number for breaking changes.

## 8. Conformance

A conforming implementation MUST:

1. Parse all examples in this specification without error
2. Produce output parseable by any other conforming implementation
3. Pass the cross-language compatibility test suite (`*/tests/cross_compat_test.*`)
4. Preserve all non-null scalar values across a text encode/decode round-trip
