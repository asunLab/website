# 为什么选择 ASON？

ASON 旨在解决 JSON 无法在不牺牲可读性或效率的前提下解决的真实问题。本页提供详细对比。

## 1. Token 效率

JSON 对数组中的每个对象都重复字段名。1000 行、5 个字段意味着 5000 个冗余 Key 字符串在网络上传输（或送入 LLM 上下文窗口）。

```json
// JSON — 约 100 Token
{
  "users": [
    { "id": 1, "name": "Alice", "active": true },
    { "id": 2, "name": "Bob",   "active": false }
  ]
}
```

```ason
// ASON — 约 35 Token（节省 65%）
[{id:int, name:str, active:bool}]:
  (1, Alice, true),
  (2, Bob,   false)
```

### 实测各载荷体积节省情况

| 载荷 | JSON 体积 | ASON Text | ASON-BIN | 文本节省 |
|------|-----------|-----------|----------|---------|
| 1000 个平面结构体（8 字段） | 121 675 B | 56 716 B | 74 454 B | **53%** |
| 100 个深层嵌套结构体（5 层） | 438 112 B | 174 611 B | 225 434 B | **60%** |

## 2. 解析性能

### 零哈希匹配（Zero Key-Hashing）

JSON 解析器必须对每一行的每个字段名进行哈希并在目标结构体中查找。1000 行 × 5 字段 = 5000 次冗余哈希操作。

ASON 解析器从 Schema 一次性建立**位置索引**（`字段 0 → id`，`字段 1 → name`，…），之后按数组下标 `O(1)` 赋值 —— 全程零哈希、零字符串比较。

### 模式驱动解析（Schema-driven Parsing）

JSON 解析器通过查看下一个字符（`"`、`t`、`f`、`[`、`{`、数字）动态推断类型，导致 CPU 分支预测频繁失败。

ASON 解析器从 Schema 得知字段类型，直接调用 `parse_int()` 或 `parse_str()` —— 无动态推断，无分支误预测。

### 基准测试（Apple M 系列，`--release` 构建）

**平面结构体（8 字段，× 1000 次）**

| 测试项 | JSON | ASON Text | ASON-BIN | BIN vs JSON |
|--------|------|-----------|----------|-------------|
| 序列化 | 2.19 ms | 1.07 ms | **0.28 ms** | **7.7×** |
| 反序列化 | 6.05 ms | 5.10 ms | **2.96 ms** | **2.0×** |

**深层嵌套结构体（5 层，× 100 次）**

| 测试项 | JSON | ASON Text | ASON-BIN | BIN vs JSON |
|--------|------|-----------|----------|-------------|
| 序列化 | 4.19 ms | 2.67 ms | **0.50 ms** | **8.4×** |
| 反序列化 | 9.37 ms | 8.55 ms | **3.72 ms** | **2.5×** |

## 3. LLM 输出质量

生成 JSON 的大模型经常出现：括号不匹配、尾逗号、字符串忘记加引号等问题。ASON 语法更宽松：

- 大多数字符串值**无需引号** —— 解析器自动去除首尾空白
- 结构**视觉线性** —— 每行一个元组，列对齐清晰
- Schema **声明式** —— 模型每次对话只需学习一次 Header

实践中意味着更少重试、更少校验错误、每次结构化响应 Token 成本更低。

## 4. 人类可读性

尽管高度压缩，ASON 依然清晰易读：

```ason
[{id:int, name:str, role:str, score:float}]:
  (1, Alice, admin,   9.5),
  (2, Bob,   viewer,  7.2),
  (3, Carol, editor,  8.8)
```

列自然对齐，Schema 行相当于表头 —— 任何读过 CSV 或 Markdown 表格的人都能立刻看懂。

## 5. 综合对比

| 特性 | JSON | ASON Text | ASON-BIN |
|------|------|-----------|----------|
| 人类可读 | ✅ | ✅ | ❌ |
| Token 高效 | ❌ | ✅ | N/A |
| Schema 分离 | ❌ | ✅ | ✅ |
| 序列化速度 | 基准 | 2× | **7–10×** |
| 反序列化速度 | 基准 | 1.2–2.5× | **2–2.5×** |
| 零拷贝字符串 | ❌ | ❌ | ✅ |
| SIMD 加速 | ❌ | ❌ | ✅ |
| LLM 友好 | ⚠️ | ✅ | ❌ |

## 何时继续使用 JSON

ASON 并非 JSON 的全面替代品。以下场景请继续使用 JSON：

- 需要临时对象结构（非同构结构体数组）
- 与要求 JSON 的外部 API 集成
- 载荷极小、频率极低，开销可以忽略

ASON 在**同构结构体数组**场景下表现最佳 —— 这也是数据管道、LLM 提示词和内部服务通信中最常见的数据模式。
